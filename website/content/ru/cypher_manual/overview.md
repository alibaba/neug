# Обзор синтаксиса

## Что такое Cypher?

Cypher — это декларативный язык запросов для графовых баз данных. Он предоставляет интуитивный и выразительный способ запроса, изменения и управления графовыми данными. Наша реализация основана на спецификации [OpenCypher](https://opencypher.org/) — открытом стандарте для языков запросов к графовым базам данных.

## Основные отличия от SQL

Если SQL предназначен для реляционных баз данных с таблицами и строками, то Cypher оптимизирован для графовых баз данных с узлами, связями и свойствами:

- **Структура**: SQL использует таблицы и JOIN-запросы; Cypher — узлы, связи и паттерны
- **Поиск по паттернам**: SQL требует явных JOIN-ов; Cypher использует синтаксис для поиска по паттернам
- **Обход графа**: SQL требует сложных JOIN-запросов для многошаговых выборок; Cypher естественным образом поддерживает обход путей
- **Читаемость**: ASCII-арт синтаксис Cypher делает графовые паттерны визуально понятными

## Что можно делать с помощью Cypher в Neug?

В Neug мы называем Cypher-запрос **Statement** (утверждение). Statement состоит из нескольких **Clauses** (клауз). Например, в следующем запросе:

```cypher
MATCH (p:person)
WHERE p.age = '29'
RETURN p.name as name;
```

Компоненты `MATCH`, `WHERE` и `RETURN` называются Clauses — это фундаментальные логические единицы для работы с графовой базой данных.

На основе OpenCypher мы определили набор Statement-синтаксиса для управления графовой базой данных Neug, включая:

### Управление схемами (DDL)

Neug в первую очередь ориентирован на сценарии работы с графовыми данными со строгой схемой, где каждая единица данных должна соответствовать заранее определённым спецификациям схемы. Это похоже на традиционный SQL; однако графовые данные включают более сложные структуры узлов и связей, которые также должны соответствовать заданным требованиям схемы.

Например, рассмотрим следующий граф схемы:

<img src="figures/modern_schema.png" alt="Modern Schema Graph" style="display: block; margin: 2em auto; max-width: 500px;">

Указанный выше граф схемы может быть создан с помощью следующих выражений:

```cypher
// Пример определения схемы
CREATE NODE TABLE person (
    name STRING,
    age INT32,
    PRIMARY KEY (name)
);

CREATE NODE TABLE software (
    name STRING,
    lang STRING,
    PRIMARY KEY (name)
);

CREATE REL TABLE knows (
    FROM person TO person,
    weight DOUBLE
);

CREATE REL TABLE created (
    FROM person TO software,
    weight DOUBLE
);
```

**Запрос, соответствующий схеме:**
В следующем запросе метка вершины `person` и метка ребра `(person-knows->person)` соответствуют схемным ограничениям, определённым выше. Узел `person` содержит свойства `age` и `name`, и свойство `age` имеет тип INT32, который можно сравнивать с константой 18. Поэтому этот запрос удовлетворяет всем ограничениям схемы и является корректным:

```cypher
MATCH (p:person)-[:knows]->(f:person)
WHERE p.age > 18
RETURN p.name, f.name;
```

**Запрос, не соответствующий схеме (вызовет ошибку):**
Метка ребра `(person-follows->person)`, указанная в этом запросе, отсутствует в схеме, что делает его некорректным и приведёт к ошибке "Table `follows` does not exist".

```cypher
MATCH (p:person)-[:follows]->(m:person)
RETURN p.name;
```

Мы определяем набор синтаксических конструкций для создания графов схем, как показано выше, который мы называем DDL (Data Definition Language). Все последующие операции обновления и запроса данных должны соответствовать спецификациям схемы, определённым текущим DDL. Мы подробно рассмотрим это в разделе [DDL](ddl_clause.md).

### Запрос данных (DQL)

Мы также определяем набор синтаксиса запросов, который может удовлетворять требованиям как Transactional Processing (TP), так и Analytical Processing (AP).

Например, вы можете запросить все треугольные паттерны в графовой базе данных, используя следующий запрос:

```cypher
MATCH (a:person)-[:created]->(b:software),
      (c:person)-[:created]->(b:software),
      (a:person)-[:knows]->(c:person)
WHERE a.name < c.name
RETURN a.name, b.name, c.name;
```

Мы называем каждый `MATCH`, `WHERE` и `RETURN` Клаузой (Clause), которые являются базовыми единицами операций с графовыми данными. Здесь операция `MATCH` в первую очередь находит все данные, составляющие треугольные паттерны, `WHERE` дополнительно фильтрует данные паттерна для обеспечения дедупликации, а операции `RETURN` выполняют проекцию имен и выводят финальные результаты. Операция `MATCH` в основном завершает сопоставление графовых паттернов, в то время как операции `WHERE`/`RETURN` в первую очередь выполняют реляционные операции, аналогичные SQL. Эти клаузы будут подробно рассмотрены в разделе [DQL](query_clauses.md).

Чтобы дополнительно обеспечить корректность операций Clause над данными, мы определили границы типов данных, поддерживаемых Neug, а также операции выражений, основанные на этих типах данных. Они будут подробно рассмотрены в разделах [Типы данных](data_types.md) и [Выражения](expression.md).

### Управление данными (DML)

Помимо DQL и DDL, Neug также поддерживает функциональность обновления данных, которую мы называем DML (Data Manipulation Language). Операции DML можно выполнять через bulk загрузки или инкрементальные обновления.

**Пример bulk импорта:**
```cypher
COPY person FROM `person.csv`;
COPY knows FROM `knows.csv`;
```

Два вышеуказанных Statement сначала выполняют bulk загрузку данных узлов с меткой `person` из файла person.csv, затем bulk загрузку данных ребер с меткой `person-[knows]->person` из файла knows.csv.

**Пример инкрементального обновления:**

Мы также предоставляем синтаксис для инкрементальной записи, позволяющий постепенно обновлять данные графа.

**Пример создания узла:**
```cypher
CREATE (p:person {name: 'Bob', age: 30});
```

**Пример создания связи:**
```cypher
MATCH (a:person {name: 'Bob'}), (b:person {name: 'marko'})
CREATE (a)-[:knows {weight: 3.0}]->(b);
```

**Пример удаления узла:**
```cypher
MATCH (p:person {name: 'Bob'})
DELETE p;
```

Мы подробно рассмотрим эти операции DML в разделе [DML section](dml_clause.md).
